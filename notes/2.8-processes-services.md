# ⚙️ Topic 2.8 — Processes and Services

## Picture this
Every program running on your Kali machine right now — even ones with no window open — is a **process**. Some of these run silently in the background 24/7, waiting for something to happen. Those are **services**.

## 1–2. Concept
- **Process** = a running instance of a program (has a unique **PID** — Process ID)
- **Service** (aka **daemon**) = a background process that starts automatically and keeps running, waiting to respond (e.g., SSH service waits for remote login attempts)

Key commands:
| Command | Meaning |
|---|---|
| `ps` | Show currently running processes |
| `ps aux` | Show ALL processes, detailed |
| `top` / `htop` | Live, updating view of processes |
| `kill PID` | Stop a process by its ID |
| `systemctl status name` | Check if a service is running |
| `systemctl start/stop name` | Start/stop a service |

## 3. Internal Working
When you run a program, Linux gives it a PID and tracks its resource use (CPU, memory). Services are managed by **systemd** — the system that starts services at boot and keeps them alive. If a service crashes, systemd can restart it automatically.

## 4. Why It Matters
Many attacks target vulnerable **services** (e.g., an outdated SSH service). Knowing how to check what's running on a machine — yours or a target's — is core recon (reconnaissance) work.

## 5. Real-World Example
A pentester scans a server and finds an old, vulnerable service still running (e.g., an unpatched FTP service). That single overlooked running service becomes the entry point for the whole attack.

## 6. Attacker Perspective
Attackers check running processes/services immediately after gaining access — to find what's running as root, what services can be abused, and what might reveal them if killed carelessly.

## 7. Defender Perspective
Defenders regularly audit running services and disable unnecessary ones — this is called **reducing attack surface** (fewer running things = fewer ways in).

## 8. Common Mistakes
- Killing the wrong process by guessing PID instead of checking first.
- Not knowing the difference between a process (temporary, one instance) and a service (persistent, background, restarts).

## 9. Practical Lab
1. `ps aux` → observe running processes
2. `top` → watch live view, press `q` to quit
3. `systemctl status ssh` → check SSH service status
4. `sudo systemctl start ssh` → start it
5. `systemctl status ssh` → confirm it's running

## 10. Key Terms
| Term | Meaning |
|---|---|
| Process | A running instance of a program |
| PID | Unique ID number for a process |
| Service/Daemon | Background process running continuously |
| systemd | Linux's system for managing services |
| Attack surface | Total number of possible entry points into a system |
