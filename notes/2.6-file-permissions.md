# 🔐 Topic 2.6 — File Permissions (chmod, chown)

## Picture this
Every file in Linux has a lock on it — who can open it, edit it, or run it. Understanding this lock system is exactly what separates a beginner from someone who understands *why* a hack even worked.

## 1–2. Concept
Every file has 3 permission types, for 3 groups of people:

**Permission types:** `r` (read), `w` (write), `x` (execute/run)
**Groups:** Owner, Group, Others

Run `ls -l` and you'll see something like: `-rwxr-xr--`
- `rwx` → Owner can read, write, execute
- `r-x` → Group can read, execute (not write)
- `r--` → Others can only read

## 3. Internal Working
Permissions are stored as numbers internally: `r=4, w=2, x=1`. Add them per group.
Example: `rwx r-x r--` = `7 5 4` → written as `chmod 754 file`

`chown user:group file` changes **who owns** the file, separate from permissions.

## 4. Why It Matters — this is your reminder topic
**SUID vs chmod 777** (you flagged this in Phase 1 — let's nail it):
- `chmod 777` = file is open to everyone for read/write/execute — a permissions problem.
- **SUID** (Set User ID) = a special bit that makes a program run **as its owner** (often root) no matter who runs it — an execution-identity problem, not an openness problem.
A file can be `chmod 777` and have no SUID, or be very restricted (`700`) and still have SUID. They solve different questions: "who can access it" vs "whose identity does it run as."

## 5. Real-World Example
A misconfigured SUID binary owned by root lets a low-privilege pentester run it and *temporarily become root* — this is a classic **privilege escalation** technique.

## 6. Attacker Perspective
Attackers actively hunt for SUID binaries (`find / -perm -4000`) after gaining low-level access, because one misconfigured SUID file can mean full system control.

## 7. Defender Perspective
Defenders regularly audit SUID files and remove unnecessary ones — fewer SUID binaries means fewer privilege escalation paths.

## 8. Common Mistakes
- Confusing `chmod 777` (open permissions) with SUID (identity-based execution) — you already caught this once, good instinct to double check.
- Using `chmod 777` as a "quick fix" — it's a major security risk, never used in real environments.

## 9. Practical Lab
1. `touch test.sh`
2. `ls -l test.sh` → note current permissions
3. `chmod 755 test.sh` → `ls -l` → confirm change
4. `chmod +x test.sh` → makes it executable
5. `chown $(whoami) test.sh` → confirm ownership

## 10. Key Terms
| Term | Meaning |
|---|---|
| chmod | Change file permissions |
| chown | Change file owner |
| SUID | Special bit making a file run as its owner's identity |
| Privilege escalation | Gaining higher access than originally granted |
